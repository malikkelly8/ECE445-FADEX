## Entry 2: Skeleton Code Test Planning

**Date:** 4/14/26

**Objective:** Define skeleton code tests that could verify basic hardware functionality before full system integration.

**Work Completed:**  
The team discussed a set of small ESP32 test programs for hardware bring-up. These included an ESP32 programming test using a blinking output pin, PWM output for the UV LED driver, ADC input for the photodiode circuit, motor driver tests for the pumps, and HX711/load-cell calibration tests. The goal was to confirm that each circuit could be controlled or read by the ESP32 before merging everything into the final firmware. These tests would also produce simple evidence such as oscilloscope waveforms, serial monitor readings, and visible pump activation.

**Results / Decisions:**  
The team decided that subsystem skeleton code would be safer and easier to debug than immediately building one large program. Each test would isolate one subsystem, making it easier to identify whether a problem came from wiring, code, power, or hardware. This approach became important for later debugging when pump pins, load-cell readings, and UV/photodiode behavior needed to be verified individually.

**Next Step:**  
Write or adapt skeleton code for ESP32 GPIO, pump motor drivers, load cells, UV LED PWM, and photodiode ADC readings.
